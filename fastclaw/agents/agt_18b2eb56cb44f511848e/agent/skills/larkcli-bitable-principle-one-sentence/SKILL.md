# larkcli-bitable-principle-one-sentence

## 适用场景
- 用户要求批量清洗飞书论文表中的`原理一句话`字段。
- 默认处理与**手势/手语识别、连续手语识别（CSLR）、手语翻译（SLT）**相关记录。
- 若用户已明确给出“当前处理清单”，则以清单为准，不因主题过滤规则跳过。
- 目标是把方法概括为一句可检索、可比较的“机制句”。

## 强约束
1. **长度**：30-50字。
2. **结构**：必须包含“方法类型 + 任务 + 关键机制”。
3. **动作词**：必须出现明确机制词，例如：对齐、融合、蒸馏、约束、重建、解码、交互、建模、选择、增强。
4. **禁止内容**：
   - 不要写成泛泛而谈的宣传语。
   - 不要只说“提出新方法/创新框架”。
   - 不要堆叠多个模块名而没有机制。

## 推荐句式
- `基于XX的YY方法，通过ZZ机制完成AA任务。`
- `一种XX式YY框架，借助ZZ对AA进行建模/对齐/解码。`
- `面向AA任务的XX方法，通过ZZ增强/约束/融合信息。`

## 证据优先级
1. PDF 方法段落、图1/总览图、方法标题
2. 记录内已有“方法概述”“贡献点”
3. 摘要中的方法描述

## 常见坑位
1. **只写任务不写机制**：如“用于手语识别的模型”不够。
2. **只写模块不写方法类型**：如“包含编码器、解码器、注意力模块”不够。
3. **机制抽象过度**：要能看出它到底怎么做，而不是抽象概括成“提升性能”。
4. **主题不相关**：无关论文不要强行套句。

## 批量实施SOP（建议）
1. **先小样后批量**：先处理 3-5 条并回读核验，再全量执行。
2. **结构闸门**：写回前逐条检查是否同时包含“方法类型 + 任务 + 关键机制”。
3. **长度闸门**：严格控制在 30-50 字；超长压缩，过短补齐机制信息。
4. **机制闸门**：必须出现可识别动作词（如对齐/融合/蒸馏/约束/重建/解码等）。
5. **证据闸门**：优先方法段与总览图；证据不足则留空并进入人工清单。

## 写回后校验
- 抽样回读：是否泛化空话、是否偏题、是否缺失机制词。
- 若出现“只写任务不写机制”或“只写模块名”，立即重写该条。

## 输出要求
- 直接输出可写回的一句话。
- 若无法提炼出清晰机制，进入人工补录清单，不要硬造。
